# Harness — 에이전트 세션·컨텍스트 전달·실행 모델

> 멀티 에이전트 하네스의 **세션 모델**, **컨텍스트 전달 방식**, **실행 순서**를 시퀀스 다이어그램과 함께 정리.
> 기준: 2026-06-23, Stage 1 완료(코어 일반화 + 3층 triage).

---

## 0. 핵심 질문 3가지 — 한 줄 답

| 질문 | 답 |
|---|---|
| **1. 각 에이전트는 별도 세션인가?** | **예.** 모든 서브에이전트는 격리된 자체 컨텍스트 윈도우에서 동작 (fork 제외) |
| **2. 컨텍스트는 어떻게 전달되나?** | **명시적 프롬프트 주입.** 오케스트레이터가 필요한 정보만 골라 다음 에이전트의 입력 프롬프트로 직렬화 |
| **3. 절차적 파이프라인인가?** | **하이브리드.** 게이트는 순차(차단형), 리뷰는 병렬(팬아웃-팬인), 전체는 오케스트레이터가 지휘 |

## 1. 세션 모델

### 1-1. 세션 격리 원칙

각 에이전트 호출은 **독립된 컨텍스트 윈도우**를 가진다. 다른 에이전트가 본 것을 자동으로 보지 못한다.

```
┌─────────────────────────────────────────────────────────┐
│  Orchestrator 세션 (메인 컨텍스트)                       │
│  - 전체 작업 흐름 관리                                   │
│  - 각 에이전트 호출 결과를 메모리(JSON)로 보관           │
│  - 다음 에이전트에 필요한 정보만 골라 전달               │
└────┬────────────────────────────────────────────────────┘
     │  (Agent 도구 호출 — 프롬프트로 컨텍스트 명시 주입)
     │
     ├─▶ ┌────────────────────────────┐
     │   │  code-writer 세션 (격리)    │   ← 자기 컨텍스트만 봄
     │   │  - 작업 명세 + 도메인 룰    │
     │   │  - 결과: diff + assumptions │
     │   └────────────────────────────┘
     │       (세션 종료, 컨텍스트 소멸)
     │
     ├─▶ ┌────────────────────────────┐
     │   │  code-reviewer 세션 (격리)  │   ← writer의 다른 것은 못 봄
     │   │  - diff + 리뷰 기준         │     (오케스트레이터가 전달한 것만)
     │   │  - 결과: findings 리포트    │
     │   └────────────────────────────┘
     │       (세션 종료)
     │
     ⋮
```

### 1-2. 격리의 함의

| 좋은 점 | 주의할 점 |
|---|---|
| 한 에이전트의 환각·노이즈가 다른 에이전트로 전파 안 됨 | 오케스트레이터가 컨텍스트 큐레이션 책임 짐 |
| 각 에이전트 컨텍스트가 작아 정확도↑ | 모든 결과를 JSON으로 직렬화해야 함 |
| 병렬 호출 가능 (서로 간섭 없음) | 같은 정보가 여러 에이전트에 중복 전달될 수 있음 |
| 감사 추적 단순 (입력/출력 1:1) | 다단계 작업은 오케스트레이터 상태기계로 관리 |

### 1-3. 예외 — fork

`subagent_type: "fork"`는 **부모 컨텍스트를 그대로 상속**한다. 이 하네스에서는 사용하지 않음 (격리 깨짐).

## 2. 컨텍스트 전달 모델

에이전트 간 직접 통신은 **없다**. 모든 데이터는 오케스트레이터를 경유한다.

### 2-1. 전달 메커니즘

```
┌─────────────────────┐
│   Orchestrator     │
│   ─ State (JSON) ─  │
│   { triage: {...},  │
│     diff: "...",    │
│     review: {...},  │
│     ...             │
│   }                 │
└──┬───────────┬──────┘
   │           │
 input       output
   ▼           ▲
┌─────────────────────┐
│  서브에이전트 세션   │
│  - 입력: 프롬프트만  │
│  - 출력: 마지막 메시지│
└─────────────────────┘
```

### 2-2. 입력 프롬프트 직렬화 예시

오케스트레이터가 code-reviewer를 호출할 때:

```
[code-reviewer 입력 프롬프트]
다음 코드 변경을 정적 리뷰하라.

## 변경 명세
- 티켓: 대출 이자 재계산 반올림 버그 수정
- 대상: portfolio/loanaccount/Interest.java

## 변경 코드 (diff)
--- a/Interest.java
+++ b/Interest.java
- return amount.setScale(2, RoundingMode.HALF_DOWN);
+ return amount.setScale(4, RoundingMode.HALF_UP);

## 작성자가 명시한 가정
- HALF_UP 반올림 규약 (finance-domain-rules §3.2)
- 4째자리 정밀도

## 리뷰 기준
가독성·설계·컨벤션·성능. 보안과 규제는 다른 에이전트가 본다.

JSON으로 반환: {verdict, findings:[...], score}
```

**전달된 것**:
- 작업 의도 (티켓 설명)
- 변경 산출물 (diff)
- 이전 에이전트가 명시한 가정
- 이 에이전트의 역할·범위 (보안·규제 제외)
- 출력 스키마

**전달되지 않은 것**:
- 다른 에이전트의 중간 산출물
- 전체 코드베이스 (필요하면 에이전트가 Read 도구로 직접 조회)
- orchestrator의 내부 상태

### 2-3. 출력 직렬화

각 에이전트는 **JSON으로 최종 결과**를 반환한다. 오케스트레이터는 이를 파싱해 state에 저장.

```
[code-reviewer 출력 — orchestrator가 받는 마지막 메시지]
{
  "verdict": "approve_with_comments",
  "findings": [
    {"severity": "info", "file": "Interest.java", "line": 42,
     "rule": "naming", "msg": "변수명 'amt' → 'principalAmount' 권장"}
  ],
  "score": 0.85
}
```

## 3. 실행 순서 — 하이브리드 모델

순수 파이프라인이 아니라, **순차(게이트) + 병렬(리뷰) + 분기(라우팅)**의 조합.

### 3-1. 실행 구조 요약

```
순차 ─ triage → planner → writer → deterministic-gate
                                         │
                              ┌──────────┴──────────┐
                       (게이트 통과 시 팬아웃)
                              │
                     ┌────────┼────────┐
            병렬 ────▶│        │        │
                  reviewer  security  regulatory
                     └────────┼────────┘
                              │
                       (팬인: 모두 도착)
                              │
                              ▼
                          qa-verifier
                              │
                              ▼
                         Human Gate
                              │
                              ▼
                         audit-trail
```

| 구간 | 모드 | 이유 |
|---|---|---|
| triage → planner → writer | 순차 | 뒷 단계가 앞 단계 산출물 필요 |
| deterministic-gate | 차단형 게이트 | 실패 시 즉시 중단 (이후 호출 비용 절약) |
| reviewer + security + regulatory | **병렬** | 서로 독립, 동시 실행으로 시간 단축 |
| qa-verifier | 순차 (모든 리뷰 후) | 리뷰 결과를 입력으로 받음 |
| Human Gate | 차단형 | 사람 승인까지 대기 |

### 4-1. 다이어그램 읽는 법

- 세로선 = 각 에이전트의 **세션 수명**. 호출 시점에 시작, 응답 시점에 종료
- 가로 화살표 = **메시지 전달** (입력 프롬프트 / 출력 JSON)
- **orchestrator가 모든 메시지를 경유** — 에이전트 간 직접 연결 없음
- 팬아웃 구간 = **세 개 세션이 동시 존재**, 서로 독립
- audit-trail에는 모든 단계 결과가 영속 기록됨

## 6. 컨텍스트 큐레이션 정책

오케스트레이터는 다음 단계에 무엇을 넘길지 결정한다. 원칙:

| 정책 | 설명 |
|---|---|
| **최소 권한** | 에이전트는 자기 역할에 필요한 것만 받음 (예: reviewer에 비밀키는 전달 X) |
| **명시적 가정 전파** | writer의 `assumptions`는 reviewer·regulator에 항상 전달 (검증 대상) |
| **결과 누적** | 다음 단계는 이전 단계 **출력**을 받음 (입력 프롬프트는 재전송 X — 중복 제거) |
| **분기 결정 기록** | triage·게이트 결정은 audit-trail에 즉시 |
| **사이즈 제약** | 큰 산출물(diff)은 요약+해시로 전달, 필요 시 Read 도구로 조회 |

## 8. FAQ

**Q. reviewer가 writer의 사고 과정(thinking)을 볼 수 있나?**
A. **아니오.** writer의 thinking은 자체 세션 안에서만 존재하고 종료 시 소멸. reviewer는 writer의 **최종 출력 JSON**만 받음.

**Q. 같은 에이전트를 여러 번 호출하면 이전 호출을 기억하나?**
A. **아니오.** 각 호출은 새 세션. 이전 호출 결과가 필요하면 orchestrator가 다음 호출 프롬프트에 명시 포함.

**Q. 병렬 리뷰 3개가 동시에 같은 코드를 보는데 충돌은?**
A. 각자 격리된 세션이라 충돌 없음. 단, 같은 diff를 3번 전달하므로 토큰 비용 3배.

**Q. qa-verifier가 실패하면 writer가 다시 시도하나?**
A. **재시도는 명시적 루프.** orchestrator가 실패 결과를 모아 writer를 새 세션으로 다시 호출(이전 시도 결과 + 실패 원인 포함). 자동 무한 루프는 없고, 최대 N회 후 사람 게이트.

**Q. 어느 에이전트가 코드를 실제로 수정하나?**
A. **code-writer만 Edit/Write 사용.** reviewer/auditor는 Read만. 실제 파일 변경은 단일 채널.
